# Wireframe Plan v1 — SignalForge

## Design principle
SignalForge v1 should feel like one coherent workflow, not a collection of disconnected tools.

The structure should support a clear progression:
Project → Inputs → Insights → Report → PRD → Tasks

---

## 1. Landing Page

### Layout
Top to bottom:
1. Header / navigation
2. Hero section
3. Product architecture section
4. Current module / demo preview
5. Strategic value section
6. Workflow overview
7. Final CTA
8. Footer

### Wireframe priorities
- Clear headline above the fold
- One primary CTA
- One secondary CTA
- Fast explanation of the workflow
- Visual separation between product vision and current module

### Important UX note
The user should understand the value before scrolling too much.

---

## 2. Create Project

### Layout
Top to bottom:
1. Page title and subtitle
2. Lightweight project form
3. Optional context area
4. Primary and secondary actions

### Form structure
- Project name
- Feature or initiative
- Background context
- Analysis goal

### Wireframe priorities
- Simple single-column form
- Minimal friction
- No unnecessary steps
- Strong primary CTA at the bottom

### Important UX note
This screen should feel like setup, not administration.

---

## 3. Input Workspace

### Layout
Top to bottom:
1. Page title and guidance
2. Input method helper
3. Main input area
4. Optional CSV upload area
5. Input tips
6. Primary CTA

### Preferred layout pattern
Two vertical zones:
- top guidance / support
- main working area below

### Main input area
The input area should be large and central.
This is one of the most important working surfaces in the MVP.

### Wireframe priorities
- Large editable text area
- Clear support for multiple input types
- Strong indication of what is accepted
- Analysis CTA visible without confusion

### Important UX note
The user should not wonder whether the input format is acceptable.

---

## 4. Insights View

### Layout
Top to bottom:
1. Page title and status
2. Summary insight block
3. Insight grid
4. Priority cues section
5. Primary and secondary actions

### Suggested structure
- Top summary area
- Two-column block for themes and pain points
- Full-width block for feature requests
- Full-width block for repeated signals or priorities
- Footer action row

### Wireframe priorities
- Highly scannable sections
- Clear labels
- Visually structured information
- Readability over density

### Important UX note
The screen must feel analytical and trustworthy, not decorative.

---

## 5. Report View

### Layout
Top to bottom:
1. Page title
2. Executive summary
3. Key findings
4. Top problems
5. Suggested opportunities
6. Recommended focus
7. Action row

### Suggested structure
Single-column reading layout with clean content blocks.

### Wireframe priorities
- Easy to read
- Stakeholder-friendly
- Strong hierarchy between summary and details

### Important UX note
This page should feel presentation-ready.

---

## 6. PRD Studio

### Layout
Top to bottom:
1. Page title and draft status
2. PRD section navigation or visible blocks
3. Editable document sections
4. Action row

### Suggested structure
A document-style layout with stacked sections:
- Problem statement
- Goals
- Target users
- Scope
- Non-goals
- Success metrics
- Risks
- Open questions

### Optional layout enhancement
A sticky side navigation for PRD sections can be added later.

### Wireframe priorities
- Feels like a real working document
- Easy to scan section by section
- Editing should feel natural

### Important UX note
The user should feel they are working with a usable product document, not a generic AI blob.

---

## 7. Task Builder

### Layout
Top to bottom:
1. Page title
2. Work buckets
3. Task groups
4. User stories
5. Acceptance criteria
6. Export / copy actions

### Suggested structure
- Top-level grouped work buckets
- Tasks listed under each bucket
- Expandable or stacked story blocks
- Acceptance criteria shown with each story or task

### Wireframe priorities
- Clear grouping
- Handoff-friendly structure
- Low ambiguity

### Important UX note
The user should be able to imagine sharing this output with engineering immediately.

---

## 8. Export / Copy Layer

### Layout
This does not need to be a full page in v1.
It can appear as:
- action buttons
- copy blocks
- export-ready text areas

### Wireframe priorities
- Simple
- low-friction
- available near output sections

### Important UX note
Copying output should feel immediate and reliable.

---

## Cross-screen navigation model

### Primary nav
- Project
- Inputs
- Insights
- Report
- PRD
- Tasks

### Navigation behavior
- Current step should always be visible
- Previous steps should be accessible
- Future steps should be visible but disabled until ready if needed

---

## Global layout principles
- Use generous spacing
- Prefer clear blocks over dense tables
- Keep page actions near the bottom and top when helpful
- Use enterprise-style visual hierarchy
- Keep the workflow linear and easy to follow

## Layout principle
The product should reduce decision fatigue by making the next step obvious at every stage.
